# OmniFocus CLI-MCP: CLI Layer Design

The fork inherits a working MCP server. This doc scopes the **CLI layer** — a Commander-based wrapper that exposes `OmniFocusClient` as the `of` command, sharing the same TypeScript codebase as the MCP server.

> **Status:** Implementation Complete (Phase 1-7). CLI layer is fully functional with functional parity with v1, codegen toolchain for command emission, and support for specialized features like indented outlines and natural dates. **Note:** The CLI currently exceeds the MCP server's surface area, serving as the authoritative layer for macOS-native workflows.

## Goals

- `of <command>` achieves **functional parity with v1** for every command in `legacy/of-cli-reference.md`. "Functional parity" = identical command names, flags, defaults, exit semantics, and JSON schemas, with documented deltas in Appendix A for output formatting where v1 used emoji/locale-dependent rendering.
- **Authoritative macOS Workflow:** The CLI serves as the primary interface for advanced macOS scripting, providing features (e.g., Quick Entry, full sync, forecast) not yet exposed via the MCP layer.
- The public CLI contract (commands, flags, output schemas, exit codes, stream routing) is defined in §"Public CLI Contract" and §Appendix A. Generated from the contract; not the other way around.
- CLI commands are **codegen-emitted** from `OmniFocusClient` signatures + `scripts/codegen/cli-metadata.ts` (the override sidecar). The sidecar is the explicit metadata model; TS signatures alone are insufficient and were never claimed to be.
- Both `of` (CLI) and `omnifocus-mcp-server` (MCP) ship from one `npm install`.
- **Primary use case:** PAI agents in hooks/scripts/pipes consume `of` via `--json` and exit codes (per memory `G3pAYE24J`: "MCP for sessions, CLI for hooks/scripts/pipes"). Human-readable output is a documented secondary surface.

## Public CLI Contract

This section defines every behavior an external consumer (shell script, agent, human) can depend on. Anything not listed here is implementation detail and may change.

### Canonical command grammar

**v1 grammar is canonical.** Commands are **verb-first** with subject following: `of <verb> [subject] [args]`.

```
✅ CANONICAL (v1):                  ❌ NON-CANONICAL (do not use):
of list inbox                       of tasks list
of list today                       of tasks list --today
of get task abc123                  of tasks get abc123
of project complete "Work"          of projects complete "Work"
of search "report"                  of database search "report"
of perspectives                     of database perspectives
```

The full canonical surface is `legacy/of-cli-reference.md` (anchored in Appendix A) plus the deltas in §A.2. The architecture diagrams in this document use v1 grammar exclusively. Any example using resource-first spelling (`of tasks list`, `of database summary`, etc.) is a documentation bug — file an issue.

**Internal organization (not user-visible):** the codegen-emitted source files live at `src/cli/commands/{list,add,modify,complete,project,folder,tag,search,get,quick,perspectives,review,sync}.ts` — verb-grouped to mirror v1 grammar. File names are implementation detail.

### Streams

```
+---------------------+----------------+----------------+----------------+
| EVENT               | --json         | --quiet        | default        |
+---------------------+----------------+----------------+----------------+
| Success data        | stdout (JSON)  | stdout (IDs)   | stdout (human) |
| Error               | stdout (JSON)* | stderr         | stderr         |
| Preflight not ready | stderr         | stderr         | stderr         |
| Dry-run echo        | stderr         | stderr         | stderr         |
| Usage hint          | stderr         | stderr         | stderr         |
| Warnings (e.g.,     | stderr         | stderr         | stderr         |
|  --dry-run on read) |                |                |                |
+---------------------+----------------+----------------+----------------+
* In --json mode, errors go to stdout as JSON so agents that pipe `of ... | jq`
  parse a single schema. Exit code still discriminates success vs failure.
  This is the one documented divergence from v1, which sent errors to stderr
  even in --json mode (see Appendix A delta D7).
```

### JSON success schema (v1-compatible wrapper)

All `--json` success output uses a stable wrapper:

```jsonc
// LIST-shaped responses (multiple entities)
{
  "success": true,
  "tasks": [...],          // OR "projects" / "folders" / "tags" / "perspectives"
  "totalCount": 42,        // OPTIONAL: present when cheaply available (see §totalCount contract below)
  "schemaVersion": "1.0"   // NEW: agents check this to handle future changes
}

// SINGLE-entity responses (get, create, update)
{
  "success": true,
  "task": { ... },         // OR "project" / "folder" / "tag"
  "schemaVersion": "1.0"
}

// WRITE responses (create/update/complete/etc.)
{
  "success": true,
  "message": "Task created",
  "id": "abc123",
  "name": "...",           // when relevant
  "schemaVersion": "1.0"
}
```

### JSON error schema

```jsonc
{
  "success": false,
  "error": "Task abc123 not found",
  "code": "NOT_FOUND",                  // matches OmniFocusError.code
  "hint": "Try `of search` to find tasks by name",  // optional
  "schemaVersion": "1.0"
}
```

Schema discriminator: `success: true | false`. Agents MUST check this before treating output as data. (`success` field is reserved at the top level; entity payloads never include it.)

**Contract guarantees:**
- Field presence: every documented field is always present in the response (even if `null`).
- Field order: not guaranteed (JSON consumers should not depend on key order).
- Additive evolution: new optional fields may be added without bumping `schemaVersion`. Removal or type changes require a major bump.
- Brief vs Full: list endpoints return Brief by default; `--full` flag promotes to Full. `get` endpoints always return Full.

### `totalCount` contract

`totalCount` is **optional** in list responses. It is present when the underlying client method computes a total cheaply; it is omitted otherwise.

| List command | `totalCount` source | Always present? |
|---|---|---|
| `list inbox` / `today` / `flagged` | `tasks.length` of unbounded result | Yes — client method already returns full set, then CLI applies `--limit` |
| `list projects` / `folders` / `tags` | array length | Yes — same reason |
| `list forecast` | sum of forecast day buckets | Yes |
| `search` | matched results length | Yes |
| `get project-tasks` | `tasks.length` from project | Yes |
| `tag tasks` | filtered length | Yes |

In the current `OmniFocusClient`, all list methods return arrays without explicit count, and `--limit` is applied client-side. Therefore `totalCount === array.length` of the unlimited result. **For all v1-canonical list commands, `totalCount` is present.** It is documented as optional only to allow future client-side optimizations (server-side limiting that returns `(items, count)` pairs) without breaking the schema contract.

### Dry-run JSON contract

When a mutating command is invoked with `--dry-run --json`, the response uses a distinct shape:

```jsonc
{
  "success": true,
  "dryRun": true,
  "method": "createTask",          // client method that would have been called
  "args": [
    { "name": "Buy groceries", "projectId": null, "flagged": true }
  ],
  "schemaVersion": "1.0"
}
```

**Discriminator:** presence of `dryRun: true`. Agents distinguish dry-run output from real success output by checking this field.

`success` is `true` because the dry-run itself succeeded. The exit code is `0`. This is intentional: shell scripts using `--dry-run` to validate before committing can rely on `$?` working uniformly across read and dry-run write commands.

If `--dry-run` is passed to a read command (no-op), the response is the read's normal output PLUS a stderr warning (`[of] --dry-run is a no-op for read commands`). The JSON does NOT include `dryRun: true` in this case — that field signals "a write would have happened."

### NOT_FOUND normalization

`OmniFocusError.NotFoundError` (exit code 4) is raised by the **client layer**, not the executor. The current `parseExecutorError()` only recognizes not-running, permission, timeout, and generic script errors — it does NOT auto-detect "not found."

**Implementation contract:** every client method that can return "entity not found" is responsible for raising `NotFoundError` directly:

```ts
// Example (in src/omnifocus/client.ts)
async getTask(idOrArgs: string | GetTaskArgs): Promise<TaskJSON> {
  const result = await runOmniJSJson(buildGetTaskScript(idOrArgs));
  if (result === null || result.notFound) {
    throw new NotFoundError(`Task not found: ${typeof idOrArgs === "string" ? idOrArgs : idOrArgs.id}`);
  }
  return result;
}
```

This is a Phase 3 implementation task: audit every client method that resolves entities by ID/name and add NotFoundError raising. Methods affected: `getTask`, `getProject`, `getFolder`, `getTag`, `updateTask`, `updateProject`, `updateFolder`, `updateTag`, `completeTask`, `dropTask`, `deleteTask`, `markReviewed`, `moveProject`, `setTaskTags`, `appendTaskNote`, `convertTaskToProject`, `addTaskNotification`, `removeTaskNotification`, `holdProject`, `activateProject`, `reorderTask`, and any methods accepting `idOrName` parameters.

The OmniJS scripts return a sentinel `{ notFound: true }` object when entity resolution fails (or `null` for some current scripts). Client methods normalize both into `NotFoundError`. Exit-code mapping in `bin/of.ts` then routes to exit 4.

### Schema versioning policy

- `schemaVersion` is `"1.0"` for v1-CLI-compatible output.
- Additive changes (new optional fields) keep `1.0`.
- Breaking changes (removed/renamed/retyped fields, changed wrapper shape) bump to `"2.0"` and require `--schema=2` opt-in for one minor release before becoming default.
- Agents pin to a major version: `--schema=1` is the explicit form of the default.

**Failure modes for `--schema=N`:**
- `--schema=1` (default): always supported, all behavior as documented.
- `--schema=N` where N is unknown or unsupported: exit 64 (USAGE) with stderr message `Unknown schema version <N>; supported: 1`. No data emitted on stdout. This applies even before any v2 exists.
- `--schema=2` while only v1 is implemented: same as unknown — exit 64. (Once v2 exists, this becomes the opt-in path.)
- Non-numeric `--schema` value: exit 64 with usage error.

### Output mode flags

```
--json         Emit JSON to stdout. Errors emitted as JSON to stdout.
--pretty       Implies --json. Pretty-prints with 2-space indent.
               (v1 behavior: legacy/src/output.js:11 treats them as equivalent.)
--quiet, -q    On success: emit only IDs on stdout, one per line.
               On failure: same as --json error to stderr (NOT stdout, since
               -q is "I want machine-cleanable IDs"). Stderr remains active
               for errors/preflight/warnings.
```

**Precedence (deterministic):** `--quiet > --pretty > --json > human`.

If `--json` is absent and `--pretty` is alone, `--pretty` is treated as `--json --pretty` (per v1).

### Exit codes (stable)

```
0   OK
1   GENERIC error (unmapped exception; should not occur in steady state)
2   NOT_RUNNING        OmniFocus app not running
3   PERMISSION_DENIED  TCC automation perms denied
4   NOT_FOUND          Task/project/folder/tag not found
5   TIMEOUT            JXA call timed out
6   SCRIPT_ERROR       OmniJS runtime exception
7   PARTIAL_BATCH      Outline batch: some succeeded, some failed
64  USAGE              Commander usage error OR parseCliDate UsageError
                       OR outline pre-validation failure OR --stdin TTY abort.
                       All "user input is wrong" errors map here.
```

Guard test asserts every `OmniFocusError` subclass has an EXIT_CODES entry. Adding an unmapped subclass becomes a build failure.

### `--quiet` semantics (precise)

| Output type | `--quiet` behavior |
|---|---|
| List of entities | One ID per line on stdout |
| Single entity (get/create/update) | The single ID on stdout |
| Write w/o ID (e.g., `sync`, `save`) | No stdout output; exit code communicates result |
| Empty result | No stdout output; exit 0 |
| Error | Goes to stderr (not stdout); JSON wrapper if `--json` also set |
| Preflight messages | Stderr, unchanged |
| Dry-run echo | Stderr, unchanged |
| Stderr warnings (e.g., `--dry-run` on read) | Emitted unchanged |

### `--pretty` semantics

`--pretty` is **only meaningful in JSON mode**. It pretty-prints with 2-space indent. If `--json` is not also set, `--pretty` implicitly enables it. There is no "pretty human" output — human mode is already as readable as v1's human mode.

### Idempotency / retry (honest contract)

- Mutating commands are **NOT idempotent**. Re-running `of complete <id>` on an already-completed task returns success (matches v1; OmniJS no-ops gracefully).
- Re-running `of add "Task"` creates a duplicate (no client-side dedupe).
- Recommended pattern: agents use `--dry-run` first, then commit; or check existence via `of get task <id> --json | jq .success` before write.
- Outline batching: NOT atomic. On partial failure, the partial-report (exit 7) maps line numbers to created IDs so callers can resume from the failure point. No automatic retry.

### Cache behavior (CLI-specific)

- The inherited `Cache` is created and destroyed per CLI invocation. There is no persistent cache between `of` calls.
- Within a single invocation, a `get` after `list` is served from cache (saves one roundtrip).
- Preflight (`client.ping()`) does NOT populate the cache — it's a separate roundtrip that returns app version only.
- Read-after-write within the same invocation always invalidates first (already client behavior).

### Latency budget (testable)

- Cold start (`of --version`): ≤ 200ms median on Apple Silicon reference machine.
- Preflight (`client.ping()`): ≤ 100ms median.
- A CLI command that does only one client read: ≤ cold-start + preflight + client-call latency.
- These targets are testable in `tests/cli/perf.test.ts` (skipped by default, runs in CI on a designated runner).

## Codegen

**Single source of truth:** `OmniFocusClient` signatures + `scripts/codegen/cli-metadata.ts`. The generator runs **before tsc** and emits TS source that is then type-checked.

**Three artifacts from one pass:**

1. `src/cli/commands/*.ts` — Commander definitions. Header: `// @generated by scripts/codegen — do not edit by hand`.
2. `$PAI_DIR/Infrastructure/OmniFocus/cli-reference.md` and `cli-reference-llm.md`.
3. `completions/{bash,zsh,fish}` — shell completions.

### `$PAI_DIR` contract

- Codegen reads `process.env.PAI_DIR`.
- If unset, falls back to `${HOME}/.config/pai`.
- If neither path exists or is writable, codegen prints a clear warning and skips reference-doc emission (still emits commands + completions). Build does NOT fail.
- In CI: `PAI_DIR` is unset; reference docs are not emitted; CI explicitly skips the `cli-reference.md` drift check (only checks `src/cli/commands/*.ts` and `completions/*` for drift).
- `--check` mode (drift detection): same skip behavior. Reference docs are best-effort artifacts, not part of the package contract.

### Drift guard

`npm run codegen:check` exits non-zero if `git diff` is non-empty after regen for files inside the repo (`src/cli/commands/`, `completions/`). External artifacts under `$PAI_DIR` are not in the drift check.

## Output Layer

### v1-wrapper transformation

Client returns raw shapes (e.g., `Promise<TaskJSON[]>` from `listTasks`). The CLI's `formatOutput()` wraps them:

```ts
// src/cli/output.ts (sketch)
export function formatOutput(result: unknown, opts: GlobalOpts): void {
  const wrapped = wrapInV1Schema(result, opts);  // adds {success, tasks/project/..., totalCount?, schemaVersion}
  if (opts.quiet) emitIds(wrapped);
  else if (opts.json || opts.pretty) emitJson(wrapped, opts);
  else emitHuman(wrapped);
}
```

`wrapInV1Schema()` is a single function with a discriminated union of inputs (TaskJSON[], ProjectJSON[], TaskJSON, ProjectJSON, etc.) and known output shapes. It is **the only place** the v1 schema is defined; `cli-metadata.ts` references it via the `outputShape` field.

### Snapshot tests (Expansion #7)

Golden files for every (output mode × entity type) combination — 6 entity shapes × 5 modes = 30 fixtures. Located in `tests/cli/__snapshots__/output/`. CI fails on diff.

## Dry-Run (Expansion #4 — default-deny)

Wraps client with a default-deny Proxy backed by **two explicit registries**, both **emitted by codegen** from the metadata sidecar:

```ts
// src/cli/dry-run.ts (codegen-emitted at top)
export const READING_METHODS = new Set<keyof OmniFocusClient>([
  "ping", "getDatabaseSummary", "search", "dumpDatabase",
  "listTasks", "getTask", "getTodayCompletedTasks",
  "listTaskNotifications", "getTaskCount",
  "listProjects", "getProject", "getReviewQueue", "getProjectTasks",
  "listFolders", "getFolder",
  "listTags", "getTag",
  "listPerspectives", "getPerspectiveTasks",
]);

export const MUTATING_METHODS = new Set<keyof OmniFocusClient>([
  "saveDatabase", "sync",
  "createTask", "updateTask", "completeTask", "uncompleteTask",
  "dropTask", "deleteTask", "moveTasks", "duplicateTasks", "setTaskTags",
  "addTaskNotification", "appendTaskNote", "convertTaskToProject",
  "removeTaskNotification", "batchCreateTasks", "batchDeleteTasks",
  "batchCompleteTasks", "reorderTask", "quickEntry",
  "createProject", "updateProject", "completeProject", "dropProject",
  "holdProject", "activateProject", "moveProject", "deleteProject",
  "markReviewed",
  "createFolder", "updateFolder", "deleteFolder",
  "createTag", "updateTag", "deleteTag",
]);

export function dryRunClient(client: OmniFocusClient): OmniFocusClient {
  return new Proxy(client, {
    get(target, prop, receiver) {
      const value = Reflect.get(target, prop, receiver);
      if (typeof value !== "function") return value;
      const name = prop as keyof OmniFocusClient;
      if (READING_METHODS.has(name)) return value.bind(target);
      if (MUTATING_METHODS.has(name)) {
        return async (...args: unknown[]) => {
          process.stderr.write(`[dry-run] ${String(name)}(${JSON.stringify(args)})\n`);
          return { dryRun: true, method: name, args };
        };
      }
      throw new UnregisteredMethodError(
        `${String(name)} is not in READING_METHODS or MUTATING_METHODS. ` +
        `Add it to scripts/codegen/cli-metadata.ts to enable --dry-run safety.`,
      );
    },
  });
}
```

**Heuristic for codegen classification:** the metadata sidecar (Appendix B) requires every method entry to declare `category: "read" | "write"`. The generator uses this to populate the registries. There is no name-prefix heuristic; explicit declaration is mandatory. Adding a method without a metadata entry causes codegen to fail.

## Preflight

```ts
// src/omnifocus/client.ts (NEW method)
async ping(): Promise<{ ok: true; version: string }> {
  return runOmniJSJson(`(() => ({ ok: true, version: app.version }))()`);
}
```

**Caveat (acknowledging Cody #5):** `ping()` distinguishes "not running" (NotRunningError) from "permissions denied" (PermissionError) because both surface as distinct stderr patterns from osascript. It does NOT verify per-command permissions — those can theoretically differ from automation perms in TCC, but in practice OmniFocus uses one TCC entry. If a command-specific perm fails after preflight passes, the actual-call PermissionError still maps to exit 3.

```ts
// src/cli/index.ts
const PREFLIGHT_BYPASS = new Set(["completion", "help"]);  // command names

program.hook("preAction", async (thisCommand) => {
  if (thisCommand.opts().noPreflight) return;
  if (PREFLIGHT_BYPASS.has(thisCommand.name())) return;
  try {
    await client.ping();
  } catch (err) {
    formatError(err, thisCommand.opts());
    process.exit(mapErrorToExitCode(err));
  }
});
```

### Preflight bypass list (precise)

The following invocations BYPASS preflight (do NOT call `client.ping()`):

| Invocation | Why | Implementation |
|---|---|---|
| `of --help` / `-h` | Built-in; never reaches `preAction` | Commander default |
| `of --version` | Built-in; never reaches `preAction` | Commander default |
| `of completion <shell>` | No OmniFocus interaction | `PREFLIGHT_BYPASS` set |
| Any command with `--no-preflight` | User opt-out for cron/batch | `noPreflight` opt check |
| Subcommand `--help` (e.g. `of list inbox --help`) | Built-in | Commander default |

All other commands preflight by default. The ~80ms tax is documented in the latency budget.

**Rationale for `completion` bypass:** `of completion zsh > ~/.zsh/_of` is commonly run during shell setup before OmniFocus is configured / running. Failing preflight here would block first-time installation.

## Outline Parser

### Grammar (precise — addresses Cody Missing Contracts #12)

```
outline       := (line)*
line          := indent? (bullet | comment | blank)
indent        := (space | tab)+         # consistent character within file
bullet        := ('- ' | '* ') content
content       := name (' #' tag)* (' ' meta)*
meta          := key ':' value
key           := 'due' | 'defer'
value         := /\S+/                  # parsed via parseCliDate()
comment       := '#' .*                 # whole-line comment, ignored
blank         := whitespace-only line, ignored
tag           := /[A-Za-z0-9_-]+/
name          := arbitrary text up to first ' #' or ' due:' / ' defer:'
```

**Rules:**
- Indentation: tabs OR spaces, consistent within the file (Cody Ambiguity #12: "consistency means one indentation character family anywhere in the file"). Mixed tabs+spaces = pre-validation failure, exit 64.
- Indent unit: detected from first indented line. Subsequent indents must be integer multiples.
- Indent skips (e.g., level 1 → level 3): pre-validation failure, exit 64.
- Top-level item = project. Items indented 1 level = tasks of that project. Items indented 2+ levels = subtasks.
- Bullet markers: `-` or `*` (matches v1 `legacy/src/commands/add.js:69` regex `/^(\s*)[-*]\s+(.+)$/`).
- Whole-line `#` comments ignored (matches v1 `add.js:28`).
- Blank lines ignored.
- **Out of scope:** headings (`#`-prefixed without bullet), checkboxes (`- [ ]`), note blocks, project/folder targeting via inline syntax. These would all be rejected at pre-validation. Use CLI flags (`--folder`, `--sequential`) instead.

### Behavior

- **Pre-validation:** parse the entire outline before any write. Structural errors (mixed indent, indent skips, bad bullet markers, malformed `due:`) abort with exit 64 and a line number.
- **Soft limit:** > 1000 tasks emits a stderr warning. `--confirm` flag is **required** to proceed (process exits 64 if absent and stdin is TTY; in non-TTY/scripted use, `--confirm` must be provided explicitly). Cody Ambiguity #13 resolved: `--confirm` REQUIRED, no interactive prompt.
- **Chunking:** 100 tasks per `batchCreateTasks` call.
- **Stdin:** explicit `--stdin` flag required. If `process.stdin.isTTY === true`, abort with usage hint and exit 64 (no hang).
- **Failure semantics:** **best-effort, continue after chunk failure.** Each chunk is sent to `batchCreateTasks` independently; a failed chunk does NOT stop subsequent chunks. Within a single chunk, atomicity depends on OmniJS batchCreate behavior — currently OmniJS processes the chunk transactionally, so a chunk either fully succeeds or fully fails. (If OmniJS changes this, the contract here would need revision.)

- **Per-entity report shape:**
  ```jsonc
  // exit 7 (PARTIAL_BATCH)
  {
    "success": false,
    "code": "PARTIAL_BATCH",
    "created": [
      { "line": 1, "type": "project", "id": "proj_abc", "name": "Q1 Goals", "parentId": null },
      { "line": 2, "type": "task",    "id": "task_def", "name": "Task 1",   "parentId": "proj_abc" },
      { "line": 3, "type": "task",    "id": "task_ghi", "name": "Task 2",   "parentId": "proj_abc" }
    ],
    "failed": [
      { "line": 5, "type": "task",    "name": "Task X", "parentId": "proj_abc", "reason": "duplicate name in project" },
      { "line": 8, "type": "project", "name": "Q2 Goals", "parentId": null,    "reason": "OmniJS timeout" }
    ],
    "schemaVersion": "1.0"
  }
  ```

- **Line numbers reference the ORIGINAL input** (including blank lines and comments). They are stable across pre-validation and execution. Agents that strip comments before parsing must reapply them to map line numbers correctly.

- **Parent-child retry consistency:** when a parent project succeeds and its child tasks fail, the partial report includes the parent's `id` in BOTH the `created[]` entry (for the project) AND the `parentId` field of each failed child. **Caller retry strategy:**
  1. Filter input lines to those in `failed[]`.
  2. For each failed task, reuse `parentId` from the report rather than creating a new project.
  3. Re-run the filtered outline with `--no-preflight --confirm` (preflight already passed).
  
  The CLI does NOT auto-retry. Re-running the entire original outline WILL create duplicates of items already in `created[]` — there's no client-side dedupe. This is the same idempotency contract documented in §"Idempotency / retry."

- **Note on top-level project failure:** if a project itself fails, all its child tasks are marked failed with `reason: "parent project creation failed"`. Caller cannot retry just the children — the parent must succeed first. Agents should re-attempt the parent before re-attempting children.

3. **No documented schemas for `search`, `sync`, `qe`, `completion` outputs.** **Recommended resolutions:**

### Architectural decisions to make at implementation time

4. **Duplicate-name resolution rule** for `idOrName` parameters (e.g., `getProject("Work")` when two "Work" projects exist). **Recommended:** ID lookup first (always unique). If input doesn't match an ID, name lookup. On multiple name matches: raise `AmbiguousNameError` (new `OmniFocusError` subclass, exit code 8 — add to EXIT_CODES) with message listing the matching IDs. Forces the caller to disambiguate by ID.

5. **Mutual-exclusion rules** for flag combinations:
   - `--brief` vs `--full`: last-wins. Both legal at parser level.
   - `--sequential` vs `--parallel` vs `--single-actions`: mutually exclusive. Commander's `addOption().conflicts()` enforces.
   - `reorder --top --bottom --before --after`: exactly one required. Commander group with `requiredOption` semantics (or custom validator).
   - `--tag` vs `--tags`: not exclusive — `--tag` sets primary, `--tags` sets remaining. Document.

6. **`completion` runtime vs static `completions/*` precedence.** `of completion <shell>` emits live-generated completion text from current metadata; `completions/*.{bash,zsh,fish}` are pre-generated for distribution (e.g., Homebrew formula install). **Recommended:** the runtime command is authoritative; the static files are convenience artifacts that the codegen drift check ensures are equivalent. Document that users can `of completion zsh > ~/.zsh/_of` OR install from the package.

7. **`--quiet` interaction with `--dry-run --json`.** Per the precedence rule (`quiet > pretty > json > human`), `--quiet --dry-run --json` should emit only the dry-run write target's ID — but a dry-run write hasn't produced an ID. **Recommended:** in `--quiet --dry-run`, emit nothing on stdout, exit 0, and let stderr's `[dry-run]` echo provide the user-visible signal. This is consistent with the contract that `--quiet` is for ID-piping.

8. **Outline parent ID across batch boundaries.** If a parent project is created in chunk 1 and child tasks land in chunk 2, the chunker must thread the parent's freshly-created ID through to the next batch. **Recommended:** outline parser tracks a `lineNumber → createdId` map across chunks; when emitting a chunk, child tasks reference their parent via the map (not via the original input position). Implementation detail — verify in tests, not contract.

### Acknowledged structural critiques (won't be resolved by more PRD prose)

9. **"Multiple sources of truth"** (§Public CLI Contract / `src/types/omnifocus.ts` / `wrapInV1Schema()`). Cody v3 is correct that prose alone cannot resolve this. The structural fix is a machine-readable command-and-schema manifest generated from `cli-metadata.ts` (Cody's recommendation). Implementing that manifest IS Phase 2 of the codegen toolchain. Phase 1 ships without it; Phase 2 makes the contract authoritative-by-construction.

10. **Effort estimate is optimistic.** Cody v3 flags ~22-30 hrs as light for the surface this covers. Realistic: probably 35-50 hrs across the full scope, with the codegen toolchain (Phase 2) being the largest underestimate. **Treat the estimate as floor, not ceiling.**

11. **Goal/scope tension** between "v1 parity" and "+13 deltas + 7 expansions + new client methods." This is intrinsic to the project (build the v1 surface AND modernize it). Documented honestly in the deltas. Engineering should flag any new delta that emerges during implementation rather than absorbing it silently.

12. **Cody's smallest-coherent-slice recommendation** (Phase 1 = read-only `list`/`get`/`search`/`review`/`perspectives`) is preserved as a fallback option. If Phase 1 of the build encounters real friction with the full surface, retreat to the read-only slice and ship that first.

# Appendix B: Codegen Metadata Model

The override sidecar `scripts/codegen/cli-metadata.ts` is the explicit metadata source. TS signatures alone are insufficient (Cody Hidden Complexity #1: "TS method signatures do not encode CLI grammar").

## B.1 Why a sidecar (not pure JSDoc)

Several v1 commands map ONE client method to MULTIPLE CLI commands with different bound args:

```
client.listTasks({...})  →  of list inbox      (binds {inbox: true})
                            of list today      (binds {dueBefore: "today"})
                            of list flagged    (binds {flagged: true})
                            of list forecast   (binds {forecast: true, days: N})
```

JSDoc-on-method tags are inadequate for this 1-to-N case. A sidecar is the natural shape.

## B.3 JSDoc shortcut (simple methods only)

For methods that map 1:1 to one CLI command with no special handling, a JSDoc tag can substitute for a sidecar entry:

```ts
/**
 * @cli of completion <shell>
 * @cli-category read
 * @cli-description Generate shell completion scripts (bash/zsh/fish)
 */
async generateCompletion(shell: "bash" | "zsh" | "fish"): Promise<string> { ... }
```

When a method has both a sidecar entry and JSDoc tags, **the sidecar wins**. The generator warns on conflict.

## B.4 Generator rules

1. **Method without metadata = build failure.** Codegen iterates every method on `OmniFocusClient` and requires either a sidecar entry, a JSDoc tag, or an explicit opt-out (`@cli-skip`).
2. **`@cli-skip`** marks methods that are CLI-internal (e.g., `ping()`) and should not become user-visible commands. **Still requires `category: "read" | "write"` classification** for the dry-run registry — `@cli-skip` only suppresses command emission, not safety classification. Methods marked `@cli-skip` without a category fail codegen.

   ```ts
   // sidecar form (skip + category required):
   CLI_METADATA.ping = { skip: true, category: "read" };
   
   // JSDoc form (skip + category required):
   /**
    * @cli-skip
    * @cli-category read
    */
   async ping(): Promise<{ ok: true; version: string }> { ... }
   ```
3. **Validation pass:** before emission, the generator checks for:
   - Duplicate command names or aliases
   - Conflicting short flags within one command (`-d` used twice)
   - `outputShape` matches client return type (compile-checked via TS types)
   - Bound `bindArgs` keys exist in the method's args type
4. **Emission order:** registries (dry-run) → command files → reference docs → completions. Each emitter consumes the same metadata; failures abort the pipeline.

## B.5 What the metadata DOES NOT do

- Does not validate help-text quality (human review at PR time).
- Does not encode v1 deltas listed in Appendix A.2 — those are implemented in `output.ts` (wrapper) and per-flag custom parsers.
- Does not handle interactive flows. CLI is non-interactive by contract.

## B.6 Maintenance contract

When `OmniFocusClient` changes:
1. Add/modify the method in `client.ts`.
2. Add/modify the corresponding entry in `cli-metadata.ts` (or `@cli-skip`).
3. Run `npm run codegen` locally. Verify diffs in `src/cli/commands/*.ts` and external refdocs.
4. CI's `codegen:check` blocks merges that skip step 2 or 3.

This is the explicit answer to Cody Hidden Complexity #1: TS signatures alone are not the codegen input. The metadata sidecar is. The sidecar's existence and shape is the public contract for "how does the CLI surface evolve."
